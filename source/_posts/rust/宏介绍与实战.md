宏（`macro`）是 Rust 在编译期生成代码的工具。函数处理已经存在的值，宏处理的是语法片段，并在类型检查之前展开成普通 Rust 代码。理解宏之后，很多可变参数、重复样板代码、领域专用语法和自动生成实现的问题，都可以用更清晰的方式解决。

本文从常用内置宏开始，逐步介绍声明宏 `macro_rules!` 的写法、重复匹配、片段说明符、导出方式，再说明过程宏的三种形态和最小工程结构。读完后，应能写出简单可用的声明宏，并能判断什么时候应该用函数、泛型、声明宏或过程宏。

## 一、为什么需要宏

### 1.1 宏解决什么问题

Rust 宏的核心价值是：在编译期根据输入的语法片段生成代码。它常见于以下场景：

| 场景 | 函数是否适合 | 宏为什么更合适 |
|------|--------------|----------------|
| 可变参数 | 不适合，函数参数个数固定 | 宏可以匹配任意数量的语法片段 |
| 避免重复代码 | 部分适合 | 宏可以生成多个函数、`impl`、匹配分支 |
| 自定义小语法 | 不适合 | 宏接收 token，不要求先形成普通表达式 |
| 编译期包含内容 | 不适合 | 宏可在编译期读取环境变量、文件、配置 |
| 自动实现 trait | 不适合 | 过程宏可以读取结构体字段并生成实现 |

例如 `println!` 可以接收不同数量的参数：

```rust
fn main() {
    println!("hello");
    println!("name = {}", "Alice");
    println!("x = {}, y = {}", 10, 20);
}
```

如果用普通函数实现，函数签名必须提前确定参数个数和类型。宏则可以先接收 `println!("x = {}, y = {}", 10, 20)` 这样的语法片段，再展开成标准库内部的格式化代码。

### 1.2 宏和函数的区别

宏看起来像函数调用，但两者工作阶段不同：

| 对比项 | 函数 | 宏 |
|--------|------|----|
| 调用形式 | `name(args)` | `name!(tokens)` |
| 工作阶段 | 运行时调用，编译期做类型检查 | 编译期展开，再参与类型检查 |
| 输入内容 | 已经符合函数签名的值 | token、表达式、类型、语句块、标识符等语法片段 |
| 参数数量 | 固定或通过切片/迭代器间接变化 | 可以用重复匹配接收任意数量 |
| 适合场景 | 业务逻辑、算法、复用行为 | 生成代码、封装重复语法、构造 DSL |

> 注意：宏不是绕过类型系统的工具。宏展开后得到的 Rust 代码仍然必须通过所有权、借用、生命周期和类型检查。

下面的宏生成了一个表达式：

```rust
macro_rules! add_one {
    ($value:expr) => {
        $value + 1
    };
}

fn main() {
    let n = add_one!(10);
    println!("n = {}", n);
}
```

运行结果：

```text
n = 11
```

`add_one!(10)` 在编译期会展开成 `10 + 1`。之后编译器才继续检查这个表达式是否合法。

## 二、Rust 常用内置宏

Rust 标准库和编译器提供了大量常用宏。日常代码里，最常见的是输出格式化、断言调试、集合构造、编译期拼接和条件判断。

### 2.1 常用宏速查

| 宏 | 用途 | 适合场景 |
|----|------|----------|
| `println!` | 格式化并输出到标准输出 | 命令行输出、调试打印 |
| `format!` | 返回格式化后的 `String` | 构造错误消息、日志文本、提示语 |
| `vec!` | 快速创建 `Vec<T>` | 初始化集合 |
| `panic!` | 主动触发 panic | 遇到不可恢复错误或示例中演示失败 |
| `assert!` | 判断条件必须为真 | 测试、运行时不变量检查 |
| `assert_eq!` | 判断左右值相等 | 单元测试、调试比较 |
| `dbg!` | 打印表达式、文件和行号，并返回表达式值 | 临时调试 |
| `todo!` | 标记尚未实现的逻辑 | 原型开发，占位分支 |
| `unimplemented!` | 标记暂不实现的逻辑 | trait 方法或分支占位 |
| `include_str!` | 编译期读取文件为 `&'static str` | 嵌入 SQL、模板、配置片段 |
| `concat!` | 编译期拼接字面量 | 构造常量字符串 |
| `env!` | 编译期读取环境变量 | 编译时版本号、构建信息 |
| `cfg!` | 判断编译配置是否启用 | 平台或 feature 分支判断 |

### 2.2 输出与格式化

`println!` 直接输出，`format!` 返回字符串，`dbg!` 适合临时查看表达式的值。

```rust
fn main() {
    let name = "Alice";
    let age = 30;

    println!("name = {}, age = {}", name, age);

    let message = format!("用户 {name} 的年龄是 {age}");
    println!("{}", message);

    let next_age = dbg!(age + 1);
    println!("next_age = {}", next_age);
}
```

标准输出（stdout）示例：

```text
name = Alice, age = 30
用户 Alice 的年龄是 30
next_age = 31
```

标准错误（stderr）示例：

```text
[src/main.rs:10:20] age + 1 = 31
```

`dbg!` 会把调试信息输出到标准错误，并包含文件名、行号和表达式文本。因此如果运行环境只展示标准输出，就会看不到这行调试信息；在本地终端运行时，标准输出和标准错误通常会一起显示。因此它适合本地调试，不适合作为正式日志接口。

### 2.3 断言与占位

`assert!` 和 `assert_eq!` 常用于测试，也可以用于检查程序内部不变量：

```rust
fn divide(a: i32, b: i32) -> i32 {
    assert!(b != 0, "除数不能为 0");
    a / b
}

fn main() {
    let result = divide(10, 2);
    assert_eq!(result, 5);
    println!("result = {}", result);
}
```

运行结果：

```text
result = 5
```

`todo!` 和 `unimplemented!` 都会在运行时触发 panic。区别主要是表达意图：

```rust
fn parse_config(_input: &str) -> usize {
    todo!("后续实现配置解析")
}

trait Storage {
    fn save(&self, value: &str);
}

struct MemoryStorage;

impl Storage for MemoryStorage {
    fn save(&self, _value: &str) {
        unimplemented!("示例中暂不实现持久化")
    }
}
```

> 注意：`todo!` 和 `unimplemented!` 不是错误处理方式。生产代码中应尽快替换为真实实现或返回 `Result`。

### 2.4 编译期拼接与包含

`concat!` 只能拼接字面量，结果在编译期确定。下面这个例子确实可以直接写成 `"/api/v1/users"`：

```rust
const API_PATH: &str = concat!("/api/", "v1/", "users");

fn main() {
    println!("{}", API_PATH);
}
```

运行结果：

```text
/api/v1/users
```

如果字符串内容完全固定，而且写在一起更清楚，直接写一个完整字符串通常更好：

```rust
const API_PATH: &str = "/api/v1/users";
```

使用 `concat!` 的价值不在于“让字符串变短”，而在于把多个编译期片段组合成一个 `&'static str`。例如字符串由路径前缀、版本号、资源名等语义片段组成时，拆开写可以让结构更清楚，并且不会产生运行时拼接成本：

```rust
const API_PATH: &str = concat!("/api/", "v1/", "users");
```

它也常和其他编译期宏一起使用。例如 `env!` 会在编译期读取环境变量，并把结果作为字符串字面量交给 `concat!`：

```rust
const BUILD_INFO: &str = concat!("package=", env!("CARGO_PKG_NAME"));
```

> 注意：`concat!` 不能直接拼接普通变量或普通常量名，因为它接收的是字面量，或者能在宏展开阶段变成字面量的结果。需要拼接运行时变量时，应使用 `format!`，但 `format!` 会在运行时生成 `String`，不能直接用于这种 `const &str` 场景。

`include_str!` 会在编译期读取文件内容，并把内容嵌入到最终二进制中：

```rust
const README: &str = include_str!("../README.md");

fn main() {
    println!("README 长度: {}", README.len());
}
```

这段代码要求 `../README.md` 在编译时存在。路径相对于当前源文件所在位置解析。

`include_str!` 对格式和大小的要求主要有三点：

| 维度 | 要求或影响 |
| --- | --- |
| 文件格式 | 文件内容必须是合法 UTF-8，因为结果类型是 `&'static str` |
| 文件大小 | 没有一个固定的语法限制，但文件内容会进入最终二进制，过大会增加编译时间、二进制体积和内存占用 |
| 文件用途 | 适合嵌入小型文本资源，例如 SQL、模板、默认配置、说明文本 |

如果要嵌入图片、压缩包、字体等二进制文件，应使用 `include_bytes!`，它返回 `&'static [u8; N]`，不要求内容是 UTF-8：

```rust
const LOGO: &[u8] = include_bytes!("../assets/logo.png");
```

> 注意：`include_str!` 只是把文件内容原样嵌入编译结果，不会解析 JSON、TOML、Markdown 等格式。格式是否正确，需要程序自己在运行时解析，或者在构建脚本、测试中提前校验。

`env!` 在编译期读取环境变量。如果变量不存在，编译会失败：

```rust
fn main() {
    let package_name = env!("CARGO_PKG_NAME");
    println!("package = {}", package_name);
}
```

`CARGO_PKG_NAME` 是 Cargo 编译时自动提供的环境变量，适合在程序中嵌入包名、版本号等构建信息。

### 2.5 集合构造与条件判断

`vec!` 是最常见的集合构造宏：

```rust
fn main() {
    let numbers = vec![1, 2, 3];
    let zeros = vec![0; 5];

    println!("{:?}", numbers);
    println!("{:?}", zeros);
}
```

运行结果：

```text
[1, 2, 3]
[0, 0, 0, 0, 0]
```

`cfg!` 返回一个布尔值，用于判断当前编译配置：

```rust
fn main() {
    if cfg!(target_os = "linux") {
        println!("当前目标平台是 Linux");
    } else {
        println!("当前目标平台不是 Linux");
    }
}
```

`cfg!` 支持的条件写法和 `#[cfg(...)]` 基本一致。它不是只支持 `target_os`，而是支持完整的配置谓词（configuration predicate）。

先看谓词语法：

| 写法 | 含义 | 示例 |
| --- | --- | --- |
| `name` | 判断某个配置名是否启用 | `cfg!(unix)` |
| `key = "value"` | 判断配置键是否等于指定值 | `cfg!(target_os = "linux")` |
| `all(...)` | 所有条件都成立 | `cfg!(all(unix, target_pointer_width = "64"))` |
| `any(...)` | 任意条件成立 | `cfg!(any(target_os = "linux", target_os = "macos"))` |
| `not(...)` | 条件取反 | `cfg!(not(debug_assertions))` |
| `true` / `false` | 固定为真或假 | `cfg!(true)` |

再看配置项。这里不能把它理解成一个完全封闭的枚举，因为 `rustc --cfg`、Cargo feature 和其他构建工具都可以额外设置配置项。Rust Reference 中常见的编译器内置配置项如下：

| 配置项 | 示例值或写法 | 说明 |
| --- | --- | --- |
| `target_arch` | `"x86_64"`、`"aarch64"`、`"wasm32"` | 目标 CPU 架构 |
| `target_os` | `"linux"`、`"windows"`、`"macos"` | 目标操作系统 |
| `target_family` | `"unix"`、`"windows"`、`"wasm"` | 目标系统家族 |
| `unix` / `windows` | `cfg!(unix)`、`cfg!(windows)` | 常用系统家族快捷配置 |
| `target_env` | `"gnu"`、`"msvc"`、`"musl"` | ABI 或运行环境 |
| `target_abi` | `""`、`"eabihf"`、`"abi64"` | 更细的目标 ABI 信息 |
| `target_vendor` | `"apple"`、`"pc"`、`"unknown"` | 目标供应商 |
| `target_endian` | `"little"`、`"big"` | 字节序 |
| `target_pointer_width` | `"16"`、`"32"`、`"64"` | 指针宽度 |
| `target_feature` | `"sse2"`、`"avx"`、`"neon"` | 目标 CPU 特性 |
| `target_has_atomic` | `"8"`、`"16"`、`"32"`、`"64"`、`"128"`、`"ptr"` | 支持对应宽度的原子操作 |
| `panic` | `"unwind"`、`"abort"` | panic 策略 |
| `proc_macro` | `cfg!(proc_macro)` | 当前 crate 是否按过程宏 crate 编译 |
| `test` | `cfg!(test)` | 是否在测试构建中 |
| `debug_assertions` | `cfg!(debug_assertions)` | 是否启用调试断言 |
| `feature` | `cfg!(feature = "serde")` | Cargo feature 是否启用 |

因此，“完整属性”更准确地说应该分成三类：

| 来源 | 示例 | 说明 |
| --- | --- | --- |
| 编译器根据目标自动设置 | `target_os`、`target_arch`、`unix` | 最常见，跨平台代码主要用这一类 |
| 构建系统设置 | `feature = "serde"` | Cargo 会按启用的 feature 设置 |
| 用户自定义 | `my_flag`、`mode = "fast"` | 通过 `rustc --cfg` 或构建脚本传入 |

用户自定义配置可以通过编译参数传入：

```bash
rustc --cfg my_flag main.rs
rustc --cfg 'mode="fast"' main.rs
```

对应判断方式：

```rust
fn main() {
    println!("{}", cfg!(my_flag));
    println!("{}", cfg!(mode = "fast"));
}
```

> 提示：某个目标平台到底启用了哪些 `cfg`，可以用 `rustc --print cfg --target <target-triple>` 查看。例如 `rustc --print cfg --target x86_64-unknown-linux-gnu`。

> 注意：`cfg!` 只是生成布尔表达式，两边分支仍然都会参与类型检查。如果需要按平台完全排除某段代码，应使用 `#[cfg(...)]` 属性。

## 三、声明宏 `macro_rules!`

声明宏（`declarative macro`）使用 `macro_rules!` 定义。它通过模式匹配输入 token，并把匹配到的片段填入展开模板。

### 3.1 基本语法

最小结构如下：

```rust
macro_rules! macro_name {
    (pattern) => {
        expansion
    };
}
```

这里 `=> { ... }` 外层花括号属于宏分支的展开模板（transcriber），用来包住“这个分支要展开成什么”。它本身是宏定义语法的一部分。模板里面可以生成表达式、语句、item 或代码块，具体取决于你在 `{ ... }` 里面写什么。

真实写法中，`pattern` 通常包含变量捕获：

```rust
macro_rules! square {
    ($value:expr) => {
        $value * $value
    };
}

fn main() {
    println!("{}", square!(4));
}
```

运行结果：

```text
16
```

`$value:expr` 表示捕获一个表达式片段，并命名为 `value`。展开部分中的 `$value` 会被替换为调用宏时传入的表达式。

这里的 `$` 是声明宏中的“宏变量”标记，用来告诉编译器：后面的名字不是普通 Rust 变量名，而是宏匹配器里的元变量（metavariable）。如果没有 `$`，`value` 就只是普通 token，宏会尝试匹配字面上的 `value` 这个词，而不是捕获调用方传入的内容。

| 写法 | 含义 |
| --- | --- |
| `$value:expr` | 在匹配阶段捕获一个表达式，并命名为 `value` |
| `$value` | 在展开阶段使用前面捕获到的内容 |
| `value` | 普通 token，只匹配或生成字面上的 `value` |

因此，`$` 可以理解为“这里要绑定或使用一个宏变量”。它和 Rust 普通变量的 `let value = ...` 不是同一层机制：普通变量存在于运行时代码里，宏变量存在于编译期的宏匹配和展开过程中。

### 3.2 多个匹配分支

`macro_rules!` 可以包含多个分支，编译器会从上到下尝试匹配：

```rust
macro_rules! describe {
    () => {
        "没有参数"
    };
    ($name:expr) => {
        concat!("一个参数: ", $name)
    };
}

fn main() {
    println!("{}", describe!());
    println!("{}", describe!("name"));
}
```

运行结果：

```text
没有参数
一个参数: name
```

分支顺序很重要。更具体的模式通常放在前面，更宽泛的模式放在后面，避免提前被宽泛模式匹配掉。

### 3.3 常见片段说明符
简单直白的说：在声明宏的 pattern 中，片段说明符用来声明：这个位置想接收哪一类 Rust 语法片段；接收到之后，通常要把它放到宏展开代码中对应的语法位置，例如表达式位置、类型位置、模式位置、生命周期位置、item 位置等。

片段说明符决定宏变量能捕获哪类语法：

| 写法             | 捕获内容          | 典型示例                             | 使用场景                               | 后续允许的 token                     |
| -------------- | ------------- | -------------------------------- | ---------------------------------- | ------------------------------- |
| `$b:block`     | 代码块           | `{ println!("hi"); }`            | 接收一段带花括号的执行逻辑                      | 任意 token                        |
| `$e:expr`      | 表达式           | `1 + 2`、`foo()`、`Some(1)`        | 接收最常见的可求值语法                        | `=>`、`,`、`;`                    |
| `$name:ident`  | 标识符           | `user`、`User`、`parse_config`     | 生成变量名、函数名、类型名                      | 任意 token                        |
| `$i:item`      | item 项        | `fn f() {}`、`struct User;`       | 生成函数、结构体、模块、trait、impl 等顶层或块内 item | 任意 token                        |
| `$lt:lifetime` | 生命周期标注        | `'a`、`'static`                   | 生成带生命周期参数的类型或函数签名                  | 任意 token                        |
| `$lit:literal` | 字面量           | `"hello"`、`42`、`true`            | 接收字符串、数字、布尔值等字面量                   | 任意 token                        |
| `$m:meta`      | 属性内部元信息       | `derive(Debug)`、`cfg(test)`      | 生成 `#[...]` 属性                     | 任意 token                        |
| `$p:pat`       | 模式            | `Some(x)`、`Ok(value)`、`A         | B`                                 | 用在 `match`、`if let`、`let` 等模式位置 |
| `$p:pat_param` | 参数位置模式        | `Some(x)`、`Ok(value)`            | 适合后面还要跟 `                          | ` 的宏匹配场景                        |
| `$path:path`   | 路径            | `std::fmt::Debug`、`Option::Some` | 接收模块路径、类型路径、函数路径                   | `=>`、`,`、`=`、`                  |
| `$s:stmt`      | 语句            | `let x = 1`、`println!("hi")`     | 接收一条语句，常用于包装执行流程                   | `=>`、`,`、`;`                    |
| `$tt:tt`       | 单个 token tree | `abc`、`{ a + b }`、`(x, y)`       | 接收更底层的 token，常用于转发给其他宏             | 任意 token                        |
| `$t:ty`        | 类型            | `String`、`Vec<i32>`、`&str`       | 生成类型别名、函数签名、泛型约束                   | `=>`、`,`、`=`、`                  |
| `$v:vis`       | 可见性修饰符        | `pub`、`pub(crate)`、空             | 生成带不同可见性的 item                     | 见下方说明                           |

> **说明**：follow-set 是宏匹配器的“后继 token 限制”。例如 `$key:expr _` 会报错，因为 `expr` 后面只能跟 `=>`、`,` 或 `;`。`block`、`ident`、`item`、`lifetime`、`literal`、`meta`、`tt` 没有特殊后继限制。`vis` 比较特殊，通常应让它后面跟 item 开头，例如 `$v:vis fn $name:ident()`；不要把它当成普通可见性字符串随意拼接。

`expr` 是最常用的片段类型，适合接收普通表达式。`tt` 更底层，适合把语法片段继续转交给其他宏，或编写更灵活的 DSL。`pat` 和 `pat_param` 容易混淆：`pat` 可以匹配更完整的模式；`pat_param` 主要用于宏匹配器中还需要把 `|` 当成分隔符的场景。

> **选择原则**：如果只是写一个普通函数、结构体或生命周期签名，直接写 Rust 代码更清楚，不需要为了使用宏而使用 `item` 或 `lifetime`。这两个片段说明符的价值在于：宏要接收调用方提供的 item，或者宏要把生命周期标注放进生成代码的固定位置。

### 3.4 重复匹配

重复匹配的基本形式是：

```rust
$( pattern ) separator repetition
```

这只是“重复片段”本身的写法。在 `macro_rules!` 的完整分支中，外层还需要有一个 matcher 定界符，例如 `(...) => { ... }`：

```rust
macro_rules! example {
    ( $( $value:expr ),+ $(,)? ) => {
        // 展开代码
    };
}
```

这里可以拆成三层理解：

| 层级 | 代码 | 作用 |
| --- | --- | --- |
| 宏分支外层 matcher | `( ... ) => { ... }` | 描述整个宏调用应该匹配什么形状 |
| 重复片段 | `$( $value:expr )` | 表示要重复捕获的语法单元 |
| 分隔符和重复次数 | `,+` | 表示用逗号分隔，并且至少出现 1 次 |

如果还要允许最后多写一个分隔符，可以在主要重复片段后面再加一个可选重复片段。分隔符不是固定的，主要重复片段用什么分隔符，尾随分隔符通常也写成同一个 token。

```rust
$( $value:expr ),* $(,)?
```

这里的 `$(,)?` 表示“逗号这个 token 出现 0 次或 1 次”，常用于支持尾随逗号。例如 `my_vec![1, 2, 3,]` 最后的逗号就是靠它匹配的。

如果分隔符换成分号，就应该写成 `$(;)?`：

```rust
$( $value:expr );* $(;)?
```

它可以匹配 `1; 2; 3`，也可以匹配 `1; 2; 3;`。因此 `$(,)?` 不是固定模板，固定的是 `$( token )?` 这种“某个 token 可选出现一次”的结构。

常见重复符号：

| 写法 | 含义 |
|------|------|
| `*` | 重复 0 次或多次 |
| `+` | 重复 1 次或多次 |
| `?` | 重复 0 次或 1 次 |

常见组合写法：

| 写法 | 含义 |
|------|------|
| `$( $value:expr ),*` | 匹配 0 个或多个逗号分隔的表达式 |
| `$( $value:expr ),+` | 匹配 1 个或多个逗号分隔的表达式 |
| `$(,)?` | 匹配 0 个或 1 个逗号，通常用于允许尾随逗号 |
| `$(;)?` | 匹配 0 个或 1 个分号，通常用于允许尾随分号 |

例如匹配逗号分隔的表达式：

```rust
macro_rules! count_args {
    () => {
        0
    };
    ( $( $value:expr ),+ $(,)? ) => {
        {
            let mut count = 0;
            $(
                let _ = &$value;
                count += 1;
            )*
            count
        }
    };
}

fn main() {
    println!("{}", count_args!());
    println!("{}", count_args!(10, 20, 30));
    println!("{}", count_args!(10, 20, 30,));
}
```

运行结果：

```text
0
3
3
```

`()` 分支专门处理没有参数的调用，`$( $value:expr ),+` 表示捕获 1 个或多个表达式，中间用逗号分隔。后面的 `$(,)?` 允许调用时带一个尾随逗号。展开部分的 `$()*` 会对每个捕获到的 `$value` 生成一段代码。

这个示例里看起来有两层 `{}`：

```rust
( $( $value:expr ),+ $(,)? ) => {
    {
        let mut count = 0;
        // ...
        count
    }
};
```

两层花括号的作用不同：

| 层级 | 作用 |
| --- | --- |
| `=> { ... }` | 宏分支的展开模板，是 `macro_rules!` 的语法结构 |
| 内层 `{ ... }` | 真正生成到调用位置的 Rust 代码块 |

内层代码块让宏展开结果成为一个表达式，所以可以写在 `println!("{}", count_args!(10, 20, 30));` 这种表达式位置；同时它也把 `count` 这样的临时变量限制在代码块内部，避免和调用方作用域里的变量互相影响。

是否需要内层 `{}`，可以按展开结果判断：

| 场景 | 是否需要内层 `{}` | 原因 |
| --- | --- | --- |
| 展开成单个表达式 | 通常不需要 | 例如 `$value * $value` 本身就是表达式 |
| 展开成多条语句，但最终还要返回一个值 | 通常需要 | 用代码块把多条语句组合成一个表达式 |
| 展开时需要临时变量 | 通常需要 | 限制临时变量作用域，减少和调用方变量冲突 |
| 展开成语句 | 通常不需要 | 例如 `println!("hi");` 可以直接作为语句展开 |
| 展开成 item | 不需要 | 例如生成 `fn`、`struct`、`impl` 时直接输出 item |

例如 `square!` 只展开成一个表达式，所以不需要内层代码块：

```rust
macro_rules! square {
    ($value:expr) => {
        $value * $value
    };
}
```

而 `count_args!` 需要先声明 `count`、循环累加，最后返回 `count`，所以需要内层代码块把这些语句包装成一个表达式。

### 3.5 宏的作用域与导出

在同一模块中，宏通常需要先定义再使用：

```rust
macro_rules! say_hi {
    () => {
        println!("hi");
    };
}

fn main() {
    say_hi!();
}
```

如果要让宏在 crate 外部使用，可以加 `#[macro_export]`：

```rust
#[macro_export]
macro_rules! say_hi {
    () => {
        println!("hi");
    };
}
```

被 `#[macro_export]` 导出的宏会放到 crate 根路径下，外部通常通过 `crate_name::say_hi!()` 调用。

> 注意：导出宏时，如果展开代码需要引用当前 crate 内部路径，通常使用 `$crate::path::to::item`，避免调用方模块路径不同导致解析失败。

## 四、声明宏实战

### 4.1 实现简化版 `my_vec!`

下面实现一个简化版 `vec!`。它接收任意数量的表达式，并依次 `push` 到 `Vec` 中。

```rust
macro_rules! my_vec {
    ( $( $value:expr ),* $(,)? ) => {
        {
            let mut temp = Vec::new();
            $(
                temp.push($value);
            )*
            temp
        }
    };
}

fn main() {
    let numbers = my_vec![1, 2, 3];
    let names = my_vec!["Alice", "Bob", "Carol",];

    println!("{:?}", numbers);
    println!("{:?}", names);
}
```

运行结果：

```text
[1, 2, 3]
["Alice", "Bob", "Carol"]
```

关键点：

| 写法 | 作用 |
|------|------|
| `$( $value:expr ),*` | 匹配逗号分隔的多个表达式 |
| `$(,)?` | 允许最后多写一个逗号 |
| `temp.push($value);` | 对每个表达式生成一次 `push` |
| 外层代码块 `{ ... }` | 让宏展开后成为一个表达式，并限制临时变量作用域 |

### 4.2 实现 `hashmap!`

标准库没有内置 `hashmap!` 宏，但可以用声明宏封装常见初始化样板：

```rust
use std::collections::HashMap;

macro_rules! hashmap {
    ( $( $key:expr => $value:expr ),* $(,)? ) => {
        {
            let mut map = HashMap::new();
            $(
                map.insert($key, $value);
            )*
            map
        }
    };
}

fn main() {
    let scores = hashmap! {
        "Alice" => 95,
        "Bob" => 88,
        "Carol" => 92,
    };

    println!("{:?}", scores.get("Alice"));
    println!("{}", scores.len());
}
```

运行结果：

```text
Some(95)
3
```

这个宏使用 `=>` 作为 key/value 分隔符，因为它比逗号更能表达“映射关系”。宏模式不要求只能使用 Rust 表达式中的运算符，也可以匹配类似 `=>` 这样的 token。

### 4.3 使用 `$ident` 生成函数

`$ident` 用来捕获标识符。它适合生成变量名、函数名、结构体名等需要标识符的位置：

```rust
macro_rules! make_getter {
    ($fn_name:ident, $field:ident, $field_type:ty) => {
        fn $fn_name(&self) -> &$field_type {
            &self.$field
        }
    };
}

struct User {
    name: String,
}

impl User {
    make_getter!(name, name, String);
}

fn main() {
    let user = User {
        name: String::from("Alice"),
    };

    println!("{}", user.name());
}
```

运行结果：

```text
Alice
```

这里 `make_getter!(name, name, String)` 会在 `impl User` 内部展开成一个方法。宏展开之后，`fn name(&self) -> &String` 仍然要符合方法定义规则。

### 4.4 简单日志宏

日志宏常见的需求是接收格式化参数，再统一加上级别前缀。可以用 `$tt` 接收剩余格式化 token，然后转交给 `format!`：

```rust
macro_rules! log_info {
    ( $( $arg:tt )* ) => {
        println!("[INFO] {}", format!( $( $arg )* ));
    };
}

fn main() {
    let user_id = 42;
    log_info!("user_id = {}", user_id);
    log_info!("task finished");
}
```

运行结果：

```text
[INFO] user_id = 42
[INFO] task finished
```

`$( $arg:tt )*` 不关心内部是字符串字面量、逗号、变量名还是表达式，而是把它们作为 token 序列整体转给 `format!`。这种写法灵活，但错误信息往往来自下游宏，调试时需要结合展开结果理解。

## 五、过程宏

过程宏（`procedural macro`）和声明宏的本质差异在于执行方式：声明宏按模式替换 token，过程宏则是一段真实运行在编译期的 Rust 代码，接收 `TokenStream` 输入，返回 `TokenStream` 输出。这意味着过程宏可以做任何普通 Rust 代码能做的事——遍历字段、读取属性、校验语法、生成多个 `impl` 块——而这些是声明宏很难甚至无法实现的。

过程宏有三类：

| 类型 | 调用形式 | 输入 | 输出 | 典型场景 |
|------|----------|------|------|----------|
| 派生宏 | `#[derive(Name)]` | 被标注的结构体或枚举定义 | 追加代码，通常是 `impl Trait for Type` | 自动生成 trait 实现 |
| 属性宏 | `#[name(args)]` | 属性参数 + 被标注的 item | 替换或包装被标注的 item | 路由注册、函数包装、测试框架 |
| 函数式宏 | `name!(...)` | 宏括号内的 token | 替换整个宏调用位置的代码 | 自定义 DSL、编译期校验 |

三类过程宏的入口签名：

| 类型 | 入口签名 |
|------|----------|
| 派生宏 | `#[proc_macro_derive(Name)] pub fn f(input: TokenStream) -> TokenStream` |
| 属性宏 | `#[proc_macro_attribute] pub fn f(attr: TokenStream, item: TokenStream) -> TokenStream` |
| 函数式宏 | `#[proc_macro] pub fn f(input: TokenStream) -> TokenStream` |

### 5.1 核心工具链：syn、quote、proc-macro2

单靠标准库的 `proc_macro::TokenStream` 能做的事情非常有限——它只能把 token 流当作字符串处理，无法理解 Rust 语法结构。真实项目几乎都会引入三个配套 crate：

| crate | 作用 | 典型用法 |
|-------|------|----------|
| `syn` | 把 `TokenStream` 解析成结构化 AST | 读取结构体字段、函数签名、属性参数 |
| `quote` | 把 Rust 代码片段组装成 `TokenStream` | 生成 `impl` 块、函数体、重复代码 |
| `proc-macro2` | `proc_macro` 类型的稳定版镜像 | 在过程宏 crate 之外操作 token（辅助函数、测试） |

#### syn：解析 TokenStream 为 AST

`syn` 提供了覆盖几乎全部 Rust 语法的 AST 类型。使用时以 `parse_macro_input!` 把输入 `TokenStream` 解析成目标类型，解析失败时自动输出编译错误而不是 `panic`。

**syn 主要类型层次**

```
TokenStream
  └─ parse_macro_input! / syn::parse_str
       ├─ DeriveInput          (#[derive(...)] 标注的类型)
       │    ├─ ident: Ident
       │    ├─ vis: Visibility
       │    ├─ attrs: Vec<Attribute>
       │    ├─ generics: Generics
       │    └─ data: Data
       │         ├─ Data::Struct(DataStruct)
       │         │    └─ fields: Fields
       │         │         ├─ Fields::Named(FieldsNamed)
       │         │         │    └─ named: Punctuated<Field, Comma>
       │         │         │         └─ Field { ident, ty, attrs, vis }
       │         │         ├─ Fields::Unnamed(FieldsUnnamed)
       │         │         └─ Fields::Unit
       │         ├─ Data::Enum(DataEnum)
       │         │    └─ variants: Punctuated<Variant, Comma>
       │         │         └─ Variant { ident, fields, discriminant, attrs }
       │         └─ Data::Union(DataUnion)
       └─ ItemFn               (属性宏标注函数时)
            ├─ attrs: Vec<Attribute>
            ├─ vis: Visibility
            ├─ sig: Signature  (函数签名)
            └─ block: Box<Block>
```

---

##### DeriveInput

派生宏入口的顶层解析目标，覆盖 `struct`、`enum`、`union` 三种形式。

| 字段 | 类型 | 说明 |
|------|------|------|
| `ident` | `Ident` | 类型名标识符，如 `User`、`Status` |
| `vis` | `Visibility` | 可见性：`pub`、`pub(crate)`、或无修饰 |
| `attrs` | `Vec<Attribute>` | 类型级别的属性，`#[derive(...)]` 本身也在其中 |
| `generics` | `Generics` | 泛型参数与 `where` 约束 |
| `data` | `Data` | 枚举：`Struct`、`Enum`、`Union`，持有具体结构体或枚举数据 |

```rust
use syn::{parse_str, DeriveInput};
use quote::quote;

fn main() {
    let src = r#"
        #[derive(Debug, Clone)]
        pub struct User {
            pub name: String,
            age: u32,
        }
    "#;
    let ast: DeriveInput = parse_str(src).unwrap();

    println!("ident: {}", ast.ident);
    println!("attrs count: {}", ast.attrs.len());
    println!("generics params: {}", ast.generics.params.len());
    let vis = &ast.vis;
    println!("vis: {}", quote!(#vis));
}
```

##### Data / DataStruct / DataEnum / Variant

`data` 字段是一个枚举，通过 `match` 区分结构体、枚举和联合体：

```rust
use syn::{parse_str, DeriveInput, Data, Fields};
```

##### Fields / FieldsNamed / Field

`Fields` 是字段集合的枚举，持有字段列表；`Field` 是单个字段的完整描述。

| Fields 变体 | 说明 |
|------------|------|
| `Fields::Named(FieldsNamed)` | 具名字段 `{ x: i32, y: i32 }` |
| `Fields::Unnamed(FieldsUnnamed)` | 元组字段 `(i32, String)` |
| `Fields::Unit` | 无字段，如 `struct Marker;` |

`FieldsNamed` 的核心属性：

| 字段 | 类型 | 说明 |
|------|------|------|
| `named` | `Punctuated<Field, Comma>` | 用逗号分隔的有序字段集合，支持 `.iter()` / `.len()` |

`Field` 各字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `ident` | `Option<Ident>` | 字段名；元组字段为 `None` |

| `vis` | `Visibility` | 字段的可见性 |
| `colon_token` | `Option<Colon>` | 具名字段的 `:` 分隔符 token |

```rust
use syn::{parse_str, DeriveInput, Data, Fields};

                println!("    attrs count: {}", field.attrs.len());
            }
        }
    }
}
```

##### Type

`Type` 是 Rust 类型表达式的枚举，常用变体：

| 变体 | 说明 | 示例 |
|------|------|------|
| `Type::Path(TypePath)` | 路径类型，最常见 | `String`、`Vec<i32>`、`std::fmt::Debug` |
| `Type::Reference(TypeReference)` | 引用类型 | `&str`、`&mut Vec<u8>` |
| `Type::Tuple(TypeTuple)` | 元组类型 | `(i32, String)` |
| `Type::Array(TypeArray)` | 固定长度数组 | `[u8; 32]` |
| `Type::Slice(TypeSlice)` | 切片类型 | `[u8]` |
| `Type::ImplTrait(TypeImplTrait)` | `impl Trait` | `impl Iterator<Item=i32>` |
| `Type::TraitObject(TypeTraitObject)` | `dyn Trait` | `dyn Display` |

`TypePath` 下的 `path: Path` 是核心字段，`Path` 由 `segments: Punctuated<PathSegment, PathSep>` 组成，每个 `PathSegment` 含 `ident`（段名）和 `arguments`（泛型参数）：

```rust
use syn::{parse_str, DeriveInput, Data, Fields, Type};
use quote::quote;

fn main() {

    let ast: DeriveInput = parse_str(src).unwrap();
    if let Data::Struct(ds) = &ast.data {
        if let Fields::Named(fn_) = &ds.fields {
            for f in &fn_.named {
                if let Type::Path(tp) = &f.ty {
                    let seg = tp.path.segments.last().unwrap();
                    println!("outermost type ident: {}", seg.ident);
                    println!("full type tokens: {}", quote!(#tp));
                }
            }
        }
    }
}
```

运行结果：

```
outermost type ident: Vec
full type tokens: Vec < Option < String > >
```

> `quote!` 打印 `TokenStream` 时在尖括号前后有空格，这是正常的 token 显示形式，不影响生成的 Rust 代码。

##### Attribute / Meta

`Attribute` 代表一个 `#[...]` 属性。

| 字段 / 方法 | 说明 |
|------------|------|
| `.path()` | 返回 `&Path`，即属性名路径，如 `serde`、`allow` |
| `.path().get_ident()` | 若属性名是单段标识符则返回 `Some(&Ident)` |
| `.meta` | `Meta` 枚举，描述属性括号内的内容 |
| `.parse_args::<T>()` | 把属性括号内容解析为 `T`（需 `T: Parse`） |
| `.parse_nested_meta(cb)` | 逐条解析 `key = value` 或 `key` 形式的属性参数 |

`Meta` 的三个变体：

| 变体 | 形式 | 说明 |
|------|------|------|
| `Meta::Path(Path)` | `#[inline]` | 属性名后无括号、无 `=` |
| `Meta::List(MetaList)` | `#[derive(Debug)]` | 属性名后跟括号，`tokens` 字段含括号内容 |
| `Meta::NameValue(MetaNameValue)` | `#[doc = "..."]` | 属性名 `=` 值，`value` 字段是 `Expr` |

```rust
use syn::{parse_str, DeriveInput, Meta, Lit, Expr};

fn main() {
    let src = r#"
        #[serde(rename = "user_name")]
        #[allow(dead_code)]
        struct User { name: String }
    "#;
    let ast: DeriveInput = parse_str(src).unwrap();
    for attr in &ast.attrs {
        let path = attr.path().get_ident().map(|i| i.to_string()).unwrap_or_default();
        println!("attr path: {}", path);
        match &attr.meta {
            Meta::Path(p)    => println!("  Path({})", quote::quote!(#p)),
            Meta::List(ml)   => println!("  List, tokens = {}", ml.tokens),
            Meta::NameValue(nv) => {
                if let Expr::Lit(el) = &nv.value {
                    if let Lit::Str(ls) = &el.lit {
                        println!("  NameValue = \"{}\"", ls.value());
                    }
                }
            }
        }
    }
}
```

运行结果：

```
attr path: serde
  List, tokens = rename = "user_name"
attr path: allow
  List, tokens = dead_code
```

字段级辅助属性（如 `#[describe(skip)]`）通常用 `parse_nested_meta` 逐条解析：

> 以下为过程宏函数体内的片段，`field` 来自上层的字段迭代，不能单独运行。

```rust
// 检查字段是否有 #[describe(skip)]

    if !attr.path().is_ident("describe") { return false; }
    let mut found = false;
    let _ = attr.parse_nested_meta(|meta| {
        if meta.path.is_ident("skip") { found = true; }

});
```

|------|------|------|
| `Generics` | `params: Punctuated<GenericParam, Comma>` | 泛型参数列表，如 `<K: Eq, V>` |
| `Generics` | `where_clause: Option<WhereClause>` | `where` 约束块 |

#### quote!：从 AST 生成代码

`quote!` 宏接收类似 Rust 源码的模板，用 `#variable` 插值替换，返回 `proc_macro2::TokenStream`。它是过程宏"输出端"的核心工具。

**插值语法速查**

| 写法 | 含义 | 要求 |
|------|------|------|
| `#var` | 把 `var` 的值展开为 token | `var` 实现 `ToTokens` |
| `#(#items)*` | 对集合的每个元素重复展开，无分隔符 | `items` 可迭代，元素实现 `ToTokens` |
| `#(#items),*` | 每个元素之间插入 `,` 分隔符 | 同上 |
| `#(#items);*` | 每个元素之间插入 `;` 分隔符 | 同上 |
| `#(#a => #b),*` | 多变量同步重复 | `a`、`b` 长度必须相同 |
| `stringify!(#ident)` | 把标识符展开为字符串字面量 | `ident` 是 `Ident` |

**原生实现 `ToTokens` 的类型**（可直接用 `#` 插值）：

`Ident`、`Literal`、`TokenStream`、`proc_macro2::TokenTree`，以及所有 `syn` 的 AST 类型（`Type`、`Path`、`Expr`、`Generics` 等）。

```rust
use syn::{parse_str, Ident, Type};
use quote::quote;

fn main() {
    let name: Ident = parse_str("MyStruct").unwrap();
    let field_names: Vec<Ident> = vec![
        parse_str("foo").unwrap(),
        parse_str("bar").unwrap(),
    ];
    let field_tys: Vec<Type> = vec![
        parse_str("i32").unwrap(),
        parse_str("String").unwrap(),
    ];

    // 单值插值 #name
    let ts1 = quote! { impl #name {} };
    println!("{}", ts1);

    // 集合重复，无分隔符 #(#items)*
    let ts2 = quote! {
        struct #name {
            #(pub #field_names: #field_tys,)*
        }
    };
    println!("{}", ts2);

    // 集合重复，带 , 分隔符 #(#items),*
    let names_ref = &field_names;
    let ts3 = quote! {
        fn fields() -> &'static [&'static str] {
            &[#(stringify!(#names_ref)),*]
        }
    };
    println!("{}", ts3);
}
```

运行结果：

```
impl MyStruct { }
struct MyStruct { pub foo : i32 , pub bar : String , }
fn fields () -> & 'static [& 'static str] { & [stringify ! (foo) , stringify ! (bar)] }
```

> `quote!` 输出的 `TokenStream` 打印时在各 token 之间有空格，这是正常的展示形式，不影响编译器解析。

#### quote_spanned!：绑定 Span 的代码生成

**什么是 Span**

`Span` 是 token 在源码中的位置信息——包含文件、行、列。每个 token 都携带 `Span`，编译器用它来把报错信息（类型不匹配、trait 未实现等）定位到对应的源码行。

**quote! 与 quote_spanned! 的区别**

| 对比项 | `quote! { ... }` | `quote_spanned! { span => ... }` |
|--------|-----------------|----------------------------------|
| 生成 token 的 Span | `Span::call_site()`（宏调用处） | 指定的 `span`（如某个字段的位置） |
| 编译错误指向 | 调用该宏的那一行 | `span` 对应的源码行 |
| 生成 token 的内容 | 相同 | 相同 |
| 适用场景 | 大多数代码生成 | 需要把 trait 约束错误归因到用户字段时 |

**语法：**

```rust
quote_spanned! { span_expr => rust_code... }
```

`span_expr` 是任何实现了 `Spanned` trait 的值（`Ident`、`Field`、`Type` 等），通过 `.span()` 方法获取。

**Spanned trait**

`syn::spanned::Spanned` 为所有 AST 类型提供 `.span()` 方法：

```rust
use syn::spanned::Spanned;

let field: &syn::Field = /* ... */;
let span = field.span();        // 整个字段声明的 Span

let ty: &syn::Type = &field.ty;
let ty_span = ty.span();        // 字段类型的 Span
```

**典型用法：trait 约束不满足时，错误指向用户字段**

> 以下为过程宏函数体内的片段，`fields`、`name` 来自上层解析结果，不能单独运行。

```rust
use quote::{quote, quote_spanned};
use syn::spanned::Spanned;

// 对每个字段生成一条 Debug 断言
// 若字段类型未实现 Debug，编译错误指向该字段而非宏调用处
let field_checks = fields.iter().map(|f| {
    let ty = &f.ty;
    quote_spanned! { f.span() =>
        let _: () = {
            fn assert_debug<T: std::fmt::Debug>() {}
            assert_debug::<#ty>();
        };
    }
});

let expanded = quote! {
    impl MyTrait for #name {
        fn describe(&self) -> String {
            #(#field_checks)*
            // ...
        }
    }
};
```

若某个字段类型没有实现 `Debug`：

- 用 `quote!`：报错指向 `#[derive(MyTrait)]` 所在行，用户难以定位
- 用 `quote_spanned! { f.span() => ... }`：报错指向 `data: MyType` 那个字段，用户一眼看到原因

```rust
use syn::{parse_str, Type};
use quote::{quote, quote_spanned};
use syn::spanned::Spanned;

fn main() {
    let field_ty: Type = parse_str("i32").unwrap();
    let ty_span = field_ty.span();

    let ts_plain   = quote! { assert_debug::<#field_ty>(); };
    let ts_spanned = quote_spanned! { ty_span => assert_debug::<#field_ty>(); };

    println!("plain tokens:  {}", ts_plain);
    println!("spanned tokens:{}", ts_spanned);
    println!("(token text identical; Span metadata differs)");
}
```

运行结果：

```
plain tokens:  assert_debug :: < i32 > () ;
spanned tokens:assert_debug :: < i32 > () ;
(token text identical; Span metadata differs — errors point to different source locations)
```

> `Span` 信息不体现在 `TokenStream` 的 `Display` 输出中，但编译器在类型检查阶段会读取每个 token 的 `Span`，进而把诊断信息指向正确的源码位置。这就是为什么 token 内容相同，但报错行号完全不同。

#### proc-macro2：跨 crate 边界的 token 类型

`proc_macro::TokenStream` 只能在过程宏 crate 内部使用。如果需要把 token 操作逻辑提取到辅助函数或独立模块中，就必须使用 `proc_macro2::TokenStream`——它的 API 与 `proc_macro` 几乎相同，但没有 crate 边界限制。

`syn` 和 `quote` 内部都使用 `proc_macro2`。在过程宏入口只需做一次类型转换：

```rust
use proc_macro::TokenStream;

pub fn my_derive(input: TokenStream) -> TokenStream {
    // quote! 返回 proc_macro2::TokenStream，用 .into() 转回 proc_macro::TokenStream
    let expanded = impl_my_trait(input.into());
    expanded.into()
}

// 辅助函数可以放在任意模块，使用 proc_macro2 类型

}
```

### 5.2 编译期错误与 Span 诊断

过程宏报告错误的方式直接影响使用者的调试体验。有两种主要机制：

#### syn::Error

`syn::Error` 允许把错误绑定到具体的 token 位置（`Span`），让编译器把错误标注在正确的源码行上：

```rust
// 错误指向宏调用处
return syn::Error::new(Span::call_site(), "Describe 只支持 struct")
    .to_compile_error()
    .into();

// 错误指向某个具体 token
return syn::Error::new(field.span(), "该字段类型不支持 Debug")
    .to_compile_error()
    .into();
```

`syn::Error` 还支持合并多个错误，把所有问题一次报出而不是遇到第一个就停止：

```rust
let mut errors = Vec::<syn::Error>::new();

for field in fields.iter() {
    if /* field 有问题 */ {
        errors.push(syn::Error::new(field.span(), "字段不满足约束"));
    }
}

if !errors.is_empty() {
    let combined = errors.into_iter().reduce(|mut a, b| { a.combine(b); a }).unwrap();
    return combined.to_compile_error().into();
}
```

#### quote_spanned!

`quote_spanned!` 把生成代码的 Span 绑定到指定位置。当生成的代码本身存在类型错误时，编译器会把错误指向原始字段，而不是宏展开后的匿名位置：

```rust
// 生成一个断言：要求字段类型实现 Debug
// 如果字段类型没有实现 Debug，错误会指向该字段的定义行
let check = quote_spanned! { field.span() =>
    struct _AssertDebug where #field_ty: std::fmt::Debug;
};
```

两者的使用场景：

| 机制 | 适合场景 |
|------|----------|
| `syn::Error::new(span, msg).to_compile_error()` | 主动检查输入结构是否合法，在宏内部生成明确错误 |
| `quote_spanned! { span => ... }` | 生成的代码依赖 trait 约束，让约束不满足时的错误指向原始字段 |

> 避免在过程宏中使用 `panic!`。`panic!` 会使编译器崩溃并输出 `proc macro panicked` 错误，调试信息很差。应始终用 `syn::Error` 或 `compile_error!` 返回明确的编译错误。

### 5.3 工程结构与依赖配置

过程宏必须放在独立的库 crate 中，并在 `Cargo.toml` 里声明 `proc-macro = true`。原因是编译器在展开调用方代码之前，必须先编译并加载过程宏 crate；两者不能混在同一个 crate 里。

典型的 workspace 结构：

```text
my-project/
├── Cargo.toml          # workspace 根配置
├── my_macros/          # 过程宏 crate
│   ├── Cargo.toml
│   └── src/lib.rs
└── app/                # 调用方 crate
    ├── Cargo.toml
    └── src/main.rs
```

根目录 `Cargo.toml`：

```toml
[workspace]
members = ["my_macros", "app"]
resolver = "2"
```

过程宏 crate 的 `my_macros/Cargo.toml`，真实项目通常需要 `syn` 和 `quote`：

```toml
[package]
name = "my_macros"
version = "0.1.0"
edition = "2024"

[lib]
proc-macro = true

[dependencies]
syn  = { version = "2", features = ["full"] }
quote = "1"
proc-macro2 = "1"
```

调用方 `app/Cargo.toml`：

```toml
[package]
name = "app"
version = "0.1.0"
edition = "2024"

[dependencies]
my_macros = { path = "../my_macros" }
```

### 5.4 派生宏实战：Describe

下面实现一个 `Describe` 派生宏，为结构体生成 `describe()` 方法，输出各字段的名称和值。字段可以加 `#[describe(skip)]` 属性排除在输出之外。

#### 辅助属性

派生宏的字段级别属性（`#[describe(skip)]`）需要在 `proc_macro_derive` 中显式声明，否则编译器会把它当成未知属性报错：

```rust
// 声明 describe 是本宏的辅助属性
#[proc_macro_derive(Describe, attributes(describe))]
pub fn derive_describe(input: proc_macro::TokenStream) -> proc_macro::TokenStream {
    // ...
}
```

#### 完整实现

`my_macros/src/lib.rs`：

```rust
use proc_macro::TokenStream;
use proc_macro2::Span;
use quote::{quote, quote_spanned};
use syn::{Data, DeriveInput, Fields, Ident, parse_macro_input};
use syn::spanned::Spanned;

#[proc_macro_derive(Describe, attributes(describe))]
pub fn derive_describe(input: TokenStream) -> TokenStream {
    let ast = parse_macro_input!(input as DeriveInput);
    let name = &ast.ident;
    let name_str = name.to_string();

    // 仅支持具名字段结构体，否则在调用处生成编译错误
    let fields = match &ast.data {
        Data::Struct(data) => match &data.fields {
            Fields::Named(f) => &f.named,
            _ => {
                return syn::Error::new(Span::call_site(), "Describe 只支持具名字段结构体")
                    .to_compile_error()
                    .into();
            }
        },
        _ => {
            return syn::Error::new(Span::call_site(), "Describe 只支持 struct")
                .to_compile_error()
                .into();
        }
    };

    // 对每个未跳过的字段生成一条 push 语句
    let field_stmts = fields.iter().filter_map(|f| {
        let skip = f.attrs.iter().any(|attr| {
            attr.path().is_ident("descrbe")
                && attr.parse_args::<Ident>().map(|id| id == "skip").unwrap_or(false)
        });
        if skip {
            return None;
        }
        let field_name = f.ident.as_ref()?;
        let field_name_str = field_name.to_string();
        // quote_spanned! 让错误指向具体字段位置
        Some(quote_spanned! { f.span() =>
            parts.push(format!("{}: {:?}", #field_name_str, self.#field_name));
        })
    });

    let expanded = quote! {
        impl Describe for #name {
            fn describe(&self) -> String {
                let mut parts: Vec<String> = Vec::new();
                #(#field_stmts)*
                format!("{}{{ {} }}", #name_str, parts.join(", "))
            }
        }
    };

    TokenStream::from(expanded)
}
```

#### 调用方代码

`app/src/main.rs`：

```rust
use my_macros::Describe;

trait Describe {
    fn describe(&self) -> String;
}

#[derive(Describe)]
struct User {
    name: String,
    age: u32,
    #[describe(skip)]   // 该字段不出现在描述中
    password: String,
}

fn main() {
    let user = User {
        name: String::from("Alice"),
        age: 30,
        password: String::from("secret"),
    };
    println!("{}", user.describe());
}
```

运行结果：

```
User{ name: "Alice", age: 30 }
```

`password` 字段因为 `#[describe(skip)]` 被过滤掉，不出现在输出中。

#### 关键点解析

| 代码片段 | 作用 |
|----------|------|
| `#[proc_macro_derive(Describe, attributes(describe))]` | 声明 `describe` 是合法的辅助属性 |
| `parse_macro_input!(input as DeriveInput)` | 解析失败时自动生成编译错误 |
| `syn::Error::new(...).to_compile_error().into()` | 主动生成指向调用位置的编译错误 |
| `fields.iter().filter_map(...)` | 遍历字段并过滤掉带 `skip` 属性的字段 |
| `quote_spanned! { f.span() => ... }` | 把错误位置指向具体字段，而非宏调用处 |
| `#(#field_stmts)*` | `quote` 的重复展开，对每个字段生成一条语句 |

### 5.5 属性宏实战：log_call

属性宏接收两个 `TokenStream`：属性参数（`attr`）和被标注的 item（`item`）。下面实现 `#[log_call]`，在函数入口和出口各打印一行日志。

`my_macros/src/lib.rs` 中添加：

```rust
use proc_macro::TokenStream;
use quote::quote;
use syn::{ItemFn, parse_macro_input};

#[proc_macro_attribute]
pub fn log_call(_attr: TokenStream, item: TokenStream) -> TokenStream {
    let input_fn = parse_macro_input!(item as ItemFn);
    let fn_name     = &input_fn.sig.ident;
    let fn_name_str = fn_name.to_string();
    let fn_vis      = &input_fn.vis;
    let fn_sig      = &input_fn.sig;
    let fn_block    = &input_fn.block;   // fn_block 是 { stmt1; stmt2; expr }

    let expanded = quote! {
        #fn_vis #fn_sig {
            println!("[LOG] >> {}", #fn_name_str);
            let __result = #fn_block;    // 把原函数体作为块表达式执行并捕获返回值
            println!("[LOG] << {}", #fn_name_str);
            __result
        }
    };

    TokenStream::from(expanded)
}
```

`app/src/main.rs` 中使用：

```rust
use my_macros::log_call;

#[log_call]
fn add(a: i32, b: i32) -> i32 {
    a + b
}

fn main() {
    let result = add(3, 4);
    println!("result = {}", result);
}
```

运行结果：

```
[LOG] >> add
[LOG] << add
result = 7
```

属性宏把原函数体（`fn_block`）整体当作块表达式执行，并用临时变量 `__result` 捕获返回值，这样就能在返回前执行收尾逻辑。`fn_sig` 和 `fn_vis` 原样保留，宏对外不改变函数签名。

> `_attr` 在这里没有被使用。如果需要读取属性参数（例如 `#[log_call(level = “debug”)]`），可以用 `syn::parse::<MetaNameValue>(attr)` 等方式解析 `_attr`。

### 5.6 函数式宏实战：make_error!

函数式过程宏的输入是宏括号内的任意 token，没有固定的 Rust 语法约束，因此可以设计完全自定义的 DSL。

自定义实现`syn.Parse`解析是关键。

下面实现 `make_error!`，接受 `VariantName = 错误码` 形式的输入，生成一个带错误码的枚举类型，并自动实现 `code()` 方法和 `Display`：

```rust
make_error! {
    NetworkError = 1001,
    TimeoutError = 1002,
    AuthError    = 1003,
}
```

展开后等价于：

```rust
#[derive(Debug)]
pub enum AppError {
    NetworkError,
    TimeoutError,
    AuthError,
}

impl AppError {
    pub fn code(&self) -> u32 {
        match self {
            AppError::NetworkError => 1001,
            AppError::TimeoutError => 1002,
            AppError::AuthError    => 1003,
        }
    }
}

impl std::fmt::Display for AppError {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        match self {
            AppError::NetworkError => write!(f, "NetworkError(1001)"),
            AppError::TimeoutError => write!(f, "TimeoutError(1002)"),
            AppError::AuthError    => write!(f, "AuthError(1003)"),
        }
    }
}
```

#### 实现

自定义语法需要为输入结构实现 `syn::parse::Parse`，然后用 `parse_macro_input!` 触发解析：

`my_macros/src/lib.rs` 中添加：

```rust
use syn::{LitInt, Token, punctuated::Punctuated, parse::Parse, parse::ParseStream};

// 单条记录：VariantName = code
struct ErrorVariant {
    name: Ident,
    code: LitInt,
}

// 完整输入：多条记录以逗号分隔
struct ErrorDefinition {
    variants: Vec<ErrorVariant>,
}

impl Parse for ErrorVariant {
    fn parse(input: ParseStream) -> syn::Result<Self> {
        let name: Ident  = input.parse()?;
        input.parse::<Token![=]>()?;
        let code: LitInt = input.parse()?;
        Ok(ErrorVariant { name, code })
    }
}

impl Parse for ErrorDefinition {
    fn parse(input: ParseStream) -> syn::Result<Self> {
        let items = Punctuated::<ErrorVariant, Token![,]>::parse_terminated(input)?;
        Ok(ErrorDefinition { variants: items.into_iter().collect() })
    }
}

#[proc_macro]
pub fn make_error(input: TokenStream) -> TokenStream {
    let def = parse_macro_input!(input as ErrorDefinition);

    let names:     Vec<&Ident>  = def.variants.iter().map(|v| &v.name).collect();
    let codes:     Vec<&LitInt> = def.variants.iter().map(|v| &v.code).collect();
    let name_strs: Vec<String>  = def.variants.iter().map(|v| v.name.to_string()).collect();

    let expanded = quote! {
        #[derive(Debug)]
        pub enum AppError {
            #(#names,)*
        }

        impl AppError {
            pub fn code(&self) -> u32 {
                match self {
                    #(AppError::#names => #codes,)*
                }
            }
        }

        impl std::fmt::Display for AppError {
            fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
                match self {
                    #(AppError::#names => write!(f, "{}({})", #name_strs, #codes),)*
                }
            }
        }
    };

    TokenStream::from(expanded)
}
```

`app/src/main.rs` 中使用：

```rust
use my_macros::make_error;

make_error! {
    NetworkError = 1001,
    TimeoutError = 1002,
    AuthError    = 1003,
}

fn main() {
    let e = AppError::TimeoutError;
    println!("error = {}, code = {}", e, e.code());
}
```

运行结果：

```
error = TimeoutError(1002), code = 1002
```

#### Parse 实现的关键点

| 写法                                               | 作用                                    |
| ------------------------------------------------ | ------------------------------------- |
| `input.parse::<Ident>()?`                        | 从输入流读取一个标识符，失败则返回错误                   |
| `input.parse::<Token![=]>()?`                    | 读取并丢弃 `=` 号，不匹配则报错                    |
| `input.parse::<LitInt>()?`                       | 读取整数字面量                               |
| `Punctuated::<T, Sep>::parse_terminated(input)?` | 读取逗号分隔的列表，允许末尾有逗号                     |
| `parse_macro_input!(input as ErrorDefinition)`   | 调用 `ErrorDefinition::parse`，失败时输出编译错误 |


## 六、宏的常见陷阱与最佳实践

### 6.1 优先考虑函数和泛型

宏很强，但不应作为默认选择。只要普通函数、泛型或 trait 能清楚表达，就优先使用它们。

```rust
fn double(value: i32) -> i32 {
    value * 2
}
```

这种逻辑不需要宏。函数有更清晰的类型签名、更好的 IDE 支持、更容易测试，也更容易定位错误。

### 6.2 注意可读性和错误信息

宏错误经常出现在展开后的代码里，错误位置可能不如普通函数直观。写宏时应控制复杂度：

| 做法 | 原因 |
|------|------|
| 保持匹配分支少而明确 | 避免调用者不知道匹配了哪个分支 |
| 为常见用法写专门分支 | 让错误信息更接近调用方输入 |
| 不在宏里隐藏大量业务逻辑 | 业务规则放函数里更易测试 |
| 给导出宏写示例和测试 | 宏的调用形式比函数更难从签名看出 |

### 6.3 理解宏卫生性

Rust 宏有卫生性（hygiene）机制（代码块的方式），能避免一部分变量名冲突。例如宏内部定义的临时变量通常不会意外覆盖调用方变量：

```rust
macro_rules! make_temp {
    () => {
        {
            let temp = 10;
            temp + 1
        }
    };
}

fn main() {
    let temp = 100;
    let value = make_temp!();

    println!("temp = {}", temp);
    println!("value = {}", value);
}
```

运行结果：

```text
temp = 100
value = 11
```

但卫生性不意味着宏里所有路径都自动正确。导出宏引用 crate 内部函数时，应使用 `$crate`：

```rust
#[macro_export]
macro_rules! call_helper {
    () => {
        $crate::helper()
    };
}
```

这样无论调用方在哪个模块调用宏，都能定位到定义宏的 crate 中的 `helper`。

### 6.4 调试宏

调试宏时，可以从三个方向入手：

| 方法 | 适合场景 |
|------|----------|
| 缩小输入 | 判断是哪个分支无法匹配 |
| 拆分宏分支 | 让错误更接近具体规则 |
| 使用 `cargo expand` | 查看宏展开后的代码 |
| 给宏写测试 | 固定常见调用形式，避免后续改坏 |

`cargo expand` 是生态中常用的宏展开查看工具。它不是标准库命令，需要单独安装：

```bash
cargo install cargo-expand
cargo expand
```

> 注意：如果项目依赖较多，`cargo expand` 输出会很长。排查时优先写一个最小示例，只保留目标宏调用。

### 6.5 何时不该用宏

以下情况通常不应优先使用宏：

| 情况 | 推荐替代 |
|------|----------|
| 固定参数的普通计算 | 函数 |
| 类型差异但逻辑相同 | 泛型和 trait bound |
| 运行时配置决定行为 | 函数、枚举、策略对象 |
| 只是想少写几行代码 | 函数或小结构体 |
| 业务规则复杂且频繁变化 | 普通模块和单元测试 |

宏适合解决“语法层面重复”或“编译期生成代码”的问题，不适合把业务逻辑藏起来。

## 七、总结与选择建议

Rust 宏的核心原则是：用编译期代码生成解决函数难以表达的问题，但不要让宏替代清晰的类型设计。

| 工具 | 解决的问题 | 优点 | 代价 | 推荐场景 |
|------|------------|------|------|----------|
| 函数 | 复用运行时逻辑 | 类型签名清楚、易测试、易调试 | 参数形态固定 | 普通业务逻辑、算法、转换 |
| 泛型 | 同一逻辑支持多种类型 | 静态派发、类型安全 | 需要设计 trait bound | 容器、算法、抽象能力 |
| 声明宏 | 按模式生成代码 | 无需额外 crate，适合重复语法 | 复杂后可读性下降 | 可变参数、集合构造、少量样板生成 |
| 过程宏 | 根据 token 生成复杂代码 | 能读取类型结构，能力最强 | 需要单独 crate，调试成本高 | derive、属性注册、DSL、框架集成 |

实际写代码时，可以按下面顺序判断：

1. 能用函数清楚表达，就用函数。
2. 只是类型不同、行为相同，优先用泛型和 trait。
3. 需要可变参数、重复语法或少量代码生成，再考虑 `macro_rules!`。
4. 需要读取结构体字段、解析属性、生成 trait 实现或设计 DSL，再考虑过程宏。

宏写得好，可以减少重复并提升 API 表达力；宏写得过度，则会让错误信息、调试和维护成本显著上升。把宏用于真正需要编译期语法生成的地方，才是 Rust 中更稳妥的选择。
